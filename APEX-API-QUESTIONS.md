# Вопросы преподавателю / IT института про API Apex ВУЗ

Чтобы курсанты могли входить в бота через учётную запись Apex и чтобы данные из Apex (ФИО, группа, курс, пропуски, расписание) использовались в системе, нужна техническая возможность подключения. Ниже — **как задуматься о безопасности** и **конкретные вопросы**, которые можно задать, даже если собеседник не знаком с термином «API».

---

## 1. Безопасность: о чём думать заранее

- **Пароли не храним.** Система не должна получать и сохранять пароль от Apex. Варианты: вход через перенаправление на страницу Apex (как «Войти через ВКонтакте») с возвратом кода/токена, либо отдельный сервис института, который по запросу только проверяет логин/пароль и возвращает «да/нет» и минимальный набор данных (ФИО, группа, курс).
- **Токен вместо пароля.** Если Apex выдаёт токен (ключ доступа) после ввода логина и пароля на своей странице — мы храним только этот токен и по нему запрашиваем данные. Срок жизни токена и способ обновления нужно уточнить.
- **Минимум данных.** Запрашивать только то, что нужно: идентификатор пользователя, ФИО, группа, курс, при необходимости — список пропусков или расписание. Лишние поля не запрашивать и не хранить.
- **Защита канала.** Запросы к Apex должны идти по HTTPS. Адрес API и токены хранить в настройках сервера (переменные окружения), не в коде и не в репозитории.
- **Права доступа.** Уточнить, кто имеет право подключаться к API (только сервер приложения, с какого IP/домена), есть ли ограничения по количеству запросов.

---

## 2. Как объяснить, что вам нужно (если про API не слышали)

Можно сказать так:

> «Мы делаем электронного помощника для курсантов (наряды, расписание, задачи). Хотим, чтобы вход был по логину и паролю от портала института (Apex), а не отдельная регистрация. То есть курсант один раз вводит свои данные Apex — и мы получаем от системы института подтверждение, что такой пользователь есть, и его ФИО, группу, курс. Пароль мы не храним и не видим — проверка должна быть на стороне института. Нужно понять, есть ли у Apex возможность такого „внешнего“ входа и выдачи минимальных данных по защищённому каналу.»
