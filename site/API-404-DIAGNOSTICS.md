# Диагностика 404 на сайте ВИТЕХ

## Что означает ошибка

Сообщение **«Failed to load resource: 404 (File not found)»** по адресам вида `api/user?telegram_id=...`, `api/duties?telegram_id=...` значит: **браузер отправил запрос по какому‑то URL и сервер ответил «страница не найдена»**.

Сайт **не «берёт информацию» из одного места** — он сам формирует URL и шлёт запросы. Откуда именно:

- В коде задаётся **базовый адрес API** (`API_BASE`).
- Каждый модуль вызывает, например: `API_BASE + '/api/user'` → получается полный URL.
- Запрос уходит **по этому URL**. Если по этому адресу не запущен ваш backend или путь неправильный — сервер (или прокси) возвращает **404**.

То есть 404 говорит: **по тому адресу, куда сайт стучится, нет такого маршрута (или нет сервера)**.

---

## Что проверить по шагам

### 1. С какого адреса открыт сайт

В адресной строке браузера должно быть что‑то вроде:

- **`http://localhost:8000/site/`** или **`http://127.0.0.1:8000/site/`**

Если открыто, например:

- **`http://localhost:5500/...`** (Live Server),
- **`file:///C:/Users/.../site/index.html`**,
- или **`https://vitechbot.online/...`**,

то запросы всё равно уходят на тот адрес, который задан как API (см. п. 2). Если этот адрес не совпадает с тем, где реально крутится ваш backend — будет 404.

### 2. Какой адрес API использует сайт

- На **экране входа** внизу есть строка: **«Адрес API: …»** — это и есть тот хост, куда уходят все запросы.
- При **404** сверху страницы появляется красный баннер с текстом вроде: «Запросы идут по адресу: …» — это тот же адрес.

Для локальной разработки там должно быть: **`http://localhost:8000`** (без `/site` и без слэша в конце).

### 3. Запущен ли backend на этом адресе

Сайт ожидает, что по адресу из п. 2 работает ваш FastAPI‑сервер и обрабатывает пути вида `/api/health`, `/api/user`, `/api/duties` и т.д.

Сделайте в папке проекта:

```bash
python server.py
```

или:

```bash
uvicorn server:app --host 0.0.0.0 --port 8000
```

После запуска откройте в браузере:

**`http://localhost:8000/api/health`**

Должен вернуться JSON, например: `{"ok": true, "service": "vitechbot-api"}`.  
Если вместо этого 404 или «сайт недоступен» — backend на этом порту не работает или отдаёт не тот путь.

### 4. Откуда именно «не найдено»

- Если **404 по `http://localhost:8000/api/...`** при запущенном `server.py` — значит, запрос доходит до сервера, но маршрут не найден (проверьте, что в `server.py` есть, например, `@app.get("/api/health")`, `@app.get("/api/user")` и т.д.).
- Если в Network вкладке видно, что запрос уходит на **другой хост/порт** (например, `http://localhost:5500/api/...` или на другой домен) — значит, сайт подставляет не тот `API_BASE`. Тогда используйте кнопку **«Указать другой адрес API»** в красном баннере и введите, например, `http://localhost:8000`.

### 5. Сброс и повторная проверка

- Чтобы сбросить сохранённый «адрес API» и заново ввести ID/группу, откройте:  
  **`http://localhost:8000/site/?reset=1`**
- Либо на экране входа нажмите **«Сбросить вход»**.

После этого снова проверьте строку «Адрес API» и убедитесь, что backend запущен именно на этом адресе и по пути `/api/health` отвечает 200.

---

## Итог

- **Откуда берётся информация:** сайт сам собирает URL из `API_BASE` и пути вроде `/api/user`, `/api/duties` и шлёт туда запросы.
- **Почему «не найдено»:** по получившемуся URL либо не запущен backend, либо путь не совпадает с маршрутами в `server.py`.
- **Что сделать:** запустить `python server.py` (или uvicorn на порту 8000), открывать сайт по **`http://localhost:8000/site/`** и при 404 смотреть красный баннер и строку «Адрес API» на экране входа — по ним видно, куда именно стучится сайт и что нужно поправить (запустить сервер или указать правильный адрес API).
